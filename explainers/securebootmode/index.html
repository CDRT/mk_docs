Issue 1 of 2

Secure Boot in Setup Mode

Systems where Secure Boot has dropped from User Mode to Setup Mode, effectively disabling Secure Boot enforcement. Affects ThinkPad laptops and ThinkCentre desktops.

Known Root Causes
KB4535680
MS Update KB4535680
January 2020 update that resolved BSOD issues but unintentionally switched affected systems into Setup Mode as a side effect.
Most Common · Windows 11
MS Update KB5025885
2023 Windows 11 update related to Black Lotus / CVE-2023-24932 KEK certificate rotation. Caused Lenovo laptop firmwares to drop into Setup Mode. Predominant cause on recent Windows 11 fleets.
Manual
Manual BIOS Intervention
A user or administrator navigated into BIOS Setup and clicked "Reset to Setup Mode" intentionally or accidentally.
⚠️
BitLocker Warning — Read Before Deploying at Scale
Always suspend BitLocker before performing any of these operations. Modifying Secure Boot state changes the value measured by PCR7, triggering a BitLocker recovery prompt on the next boot. Ensure recovery keys are escrowed in Active Directory or Intune before deploying at scale.
Remediation Paths — Choose by Device Platform
Path A ThinkPad Laptops

WMI — Restore Factory Keys. Resets all Secure Boot key databases (PK, KEK, db, dbx) to factory defaults. Requires a Supervisor Password (Legacy alphanumeric only). Supported on X/T/P Gen 2+, L Gen 3+, E Gen 4+ models.

1
Verify the method is supported
Run from an elevated PowerShell prompt. If RestoreFactoryKeys appears in output, the method is available on this system.
PowerShell — Check support
# Check if Lenovo_FunctionRequest exists Get-CimInstance -Namespace "root\wmi" -ClassName "Lenovo_FunctionRequest" | ForEach-Object { if ($_.CurrentSetting -ne "") { Write-Host $_.CurrentSetting.Replace(","," = ") } }
ℹ️
Output shows RestoreFactoryKeys = No — this does not mean unavailable; it means not yet executed. If the line is entirely absent, the script method is not supported on that model. Use the manual BIOS approach instead.
2
Execute Restore Factory Keys
Replace 0000 with the actual Supervisor Password. Both commands are required.
PowerShell — Set + Save
Invoke-CimMethod -Namespace "root\wmi" ` -ClassName "Lenovo_SetFunctionRequest" ` -MethodName "SetFunctionRequest" ` -Arguments @{ parameter = "RestoreFactoryKeys,Yes,0000,ascii,us" } Invoke-CimMethod -Namespace "root\wmi" ` -ClassName "Lenovo_SaveBiosSettings" ` -MethodName "SaveBiosSettings" ` -Arguments @{ parameter = "0000,ascii,us" }
Path B ThinkCentre Desktops

SRWINx64.exe — Targeted mode switch. Sets Secure Boot directly to User Mode without resetting key databases (PK, KEK, db, dbx). Preferred approach where applicable. Supervisor Password required.

1
Download Windows BIOS Setting Tool
Download SRWINx64.exe from the BIOS download section of your model's page at pcsupport.lenovo.com. Extract the archive.
2
Run the mode switch command
Replace xxxxxx with the Supervisor Password. A reboot is required for the change to take effect.
Command Prompt / PowerShell
SRWINx64.exe /sbm user /pass xxxxxx
Validated Models
M920x  ·  M70q Gen1 / Gen4 / Gen5  ·  M75q Gen2
Neo 50q Gen4  ·  P360 Ultra  ·  P5  ·  P8 ThinkStation
Issue 2 of 2

KEK 2023 Not Delivered by Windows Update

On specific ThinkCentre and ThinkStation models, CA 2023 migration fails with Event ID 1795 in the Windows Event Log — despite systems being fully patched and Secure Boot enabled. This is a Windows Update resolution issue, not a missing Lenovo deliverable.

🔍
Root Cause
The PK thumbprint is present in Microsoft's kek_update_map.json and the correct binary (KEKUpdate_Lenovo_PK70.bin) is published. Lenovo has done its part — the signed binary was submitted to Microsoft months ago. The issue lies in how Windows Update processes this information for these specific models. The signed binary, map entry, and cryptographic chain are all valid.
Affected Models
ThinkCentre M720 Series
M720t  ·  M720s  ·  M720q
ThinkCentre M920 Series
M920t  ·  M920s
M920q  ·  M920x
ThinkStation
P330 Tiny
BIOS Family: M1U    |    Shared PK Thumbprint: F2E851A4D6C74F0671DC0DCF392C96F94AE30816

If you observe Event ID 1795 on Lenovo models not listed above, run the diagnostic script (Solution 2 below) — it reports the PK thumbprint and model info needed to identify the correct binary.
⚠️
BitLocker Warning
Injecting KEK 2023 modifies the Secure Boot state measured by PCR7. On the first reboot after the operation, BitLocker may trigger a one-time recovery prompt — the recovery key is required. The Apply-LenovoKEK.ps1 script suspends BitLocker automatically and resumes it after the operation. Ensure recovery keys are escrowed in AD or Intune before deploying at scale.
Remediation Paths
Solution 1 — Official BIOS Fix
BIOS-Level Remediation
When to use: BIOS access is available and a Supervisor Password is set on the fleet.
Requires a BIOS update that includes KEK 2023 in the Default key store (confirm applicable version with Lenovo support), plus SRWINx64.exe v1.9.4.0 or later (released 2026-03-09).
Windows BIOS Setting Tool
SRWINx64.exe /sbm reskeys /pass <SupervisorPassword>

The /sbm reskeys parameter restores all default keys to the active key store.

🔐
A Supervisor Password is mandatory. This command will not work if no password is configured.
Solution 2 — OS-Level Workaround
Manual KEK Binary Injection
When to use: No BIOS access or Supervisor Password available — or for large-scale automated remediation via Windows only.
Manually applies the Microsoft-signed binary from the OS layer. Validated cryptographically against the machine's PK — if the firmware accepts it, the procedure is confirmed sound. Works whether KEK 2023 is the only missing piece or no CA 2023 updates have been applied yet. No third-party dependencies. Admin rights required.
Find-LenovoKEK.ps1
Diagnostic Only
Reads machine PK, looks up correct binary in Microsoft's map, downloads locally. No UEFI write.
Apply-LenovoKEK.ps1
Remediation
Applies binary to UEFI KEK. Auto-suspends/resumes BitLocker. Idempotent — safe to re-run.
✅
Uses Set-SecureBootUEFI -AppendWrite — re-running on a machine where KEK 2023 is already present will not duplicate the entry.
After Remediation

Verifying the Fix

After applying either solution, confirm that KEK 2023 is present in the active KEK store.

Quick Check — PowerShell
Returns True if KEK 2023 is present in the active store. Run from an elevated prompt.
PowerShell
[System.Text.Encoding]::ASCII.GetString((Get-SecureBootUEFI -Name KEK).bytes) -match 'CA 2023'
